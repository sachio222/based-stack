# Formatting Habits

Load when writing or editing long-form pieces where formatting
choices matter.

## Dashes

Em-dashes with spaces — like this — are the workhorse. Heavy use,
for:

- Inline asides that shouldn't interrupt the sentence too hard.
- Contrast within a sentence — the old way, the new way.
- Adding a beat before a landing phrase.

**Do not** use the hyphen for this purpose. It's visually cramped.

## Parentheticals

(Like this.) Use sparingly — no more than two per paragraph — for:

- Quick clarifications that would derail the main sentence.
- Softeners on strong claims ("the closest thing to a silver bullet
  (for most cases)").
- Inside jokes or self-deprecating asides.

If you're reaching for a third parenthetical in a paragraph, the
paragraph wants to be restructured.

## Ellipses

Use "…" for:

- A trailing thought or dramatic pause. ("No, really…")
- Quoted inner monologue that trails off.

Don't use for list omissions in prose. Use prose.

## Bold

Bold the **single word or phrase** worth remembering per paragraph.
Not more. If you're bolding every third word, you've abdicated
emphasis — the reader doesn't know what matters.

Common bolding targets:

- The new term being defined.
- The counterintuitive fact that matters ("Ignore the **free**
  column").
- The rule ("Every `0.0.0.0` binding should be **deliberate**").

## Italics

Italics carry beat emphasis. "It *didn't* tell you where you are."
The italicized word is the one the reader should stress mentally.

Also used for:

- Publication or book titles.
- A word being treated as a word ("The word *available* means…").

## Headers

- Sentence case or noun phrases. Never title-case-everything.
- No emoji in headers.
- Numbered headers (1, 2, 3) are fine for sequential guides —
  Terminal Forensics uses them. Skip numbering for exploratory
  essays where order is thematic rather than procedural.

## Lists vs prose

- Use lists for **genuinely parallel items** — install commands,
  comparison tables, rule checklists.
- Use prose when voice and rhythm matter. A list of three ideas
  that should flow into each other kills the voice.

**Red flag:** if each bullet is a full sentence that could stand on
its own, the list wants to be a paragraph.

## Tables

Use tables for:

- Comparison matrices (old → new, feature A vs feature B).
- Reference data (shortcuts, flag meanings, priority levels).

Don't use tables to hide prose. If the "cell" is two sentences long,
it's not a table row.

## Code blocks

- Plain, tight, no inline comments cluttering them.
- Explain in prose above or below the block.
- Leading space inside code blocks is meaningful — don't add it for
  decoration.
- Use ` ```bash ` syntax specifier where relevant.

## Links

- In-text links are fine. No need to bracket them as "click here."
  The linked text is the pointer.
- For long URLs in a reference list, it's okay to show the raw URL
  — honesty over prettiness.
